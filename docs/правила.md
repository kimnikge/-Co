

# Правила работы ИИ‑Агента в проекте

Документ определяет границы действий ИИ‑Агента, допустимые инструменты и принципы экономии токенов.

---

# 1. Общие принципы

ИИ‑Агент работает:

- Строго в рамках текущего проекта
- Только с существующей архитектурой (SvelteKit + JSON storage)
- Без самовольного изменения стека
- Без внедрения лишней инфраструктуры

Приоритет:
1. Точность
2. Минимализм
3. Производительность
4. Безопасность

---

# 2. Что ИИ‑Агент МОЖЕТ делать

## 2.1 Разработка

- Создавать и редактировать файлы проекта
- Писать код для:
  - Svelte компонентов
  - Server endpoints
  - API
  - Логики калькулятора
  - Админ-панели
- Оптимизировать существующий код
- Исправлять баги

---

## 2.2 Архитектура

- Улучшать структуру проекта без усложнения
- Выносить конфигурации в отдельные файлы
- Предлагать рефакторинг при необходимости
- Готовить проект к масштабированию

---

## 2.3 Безопасность

- Добавлять валидацию
- Добавлять rate limiting
- Усиливать auth-flow
- Улучшать обработку ошибок

---

## 2.4 Производительность

- Уменьшать размер bundle
- Оптимизировать SSR
- Предлагать lazy loading
- Минимизировать лишние зависимости

---

# 3. Что ИИ‑Агент НЕ ИМЕЕТ ПРАВА делать

## 3.1 Запрещено менять стек

Нельзя:

- Добавлять базы данных без запроса
- Подключать ORM
- Внедрять сторонние SaaS без одобрения
- Переводить проект на другой фреймворк

---

## 3.2 Запрещено усложнять

Нельзя:

- Добавлять избыточные абстракции
- Делать enterprise-архитектуру без необходимости
- Добавлять лишние зависимости
- Делать over-engineering

---

## 3.3 Запрещено нарушать безопасность

- Нельзя хранить пароли в коде
- Нельзя логировать чувствительные данные
- Нельзя запрашивать лишние персональные данные
- Нельзя ослаблять проверку авторизации

---

# 4. Разрешённые инструменты

ИИ‑Агент может использовать:

- SvelteKit
- Node adapter
- JSON file storage
- Telegram Bot API
- ENV-переменные
- Стандартные Node модули

Дополнительные библиотеки допустимы только при необходимости и с обоснованием.

---

# 5. Экономия токенов

## 5.1 При генерации кода

- Писать только необходимый код
- Не повторять существующую структуру
- Не переписывать файл полностью, если меняется только часть
- Использовать oboe.edit_file для точечных изменений

---

## 5.2 При объяснениях

- Кратко, по делу
- Без повторения уже утверждённых решений
- Без избыточной теории

---

## 5.3 При архитектурных решениях

- Предлагать минимум вариантов
- Не генерировать большие абстрактные блоки без запроса
- Не добавлять «на будущее», если это не требуется

---

# 6. Приоритет оптимизации

Порядок при принятии решений:

1. Безопасность
2. Простота
3. Производительность
4. Масштабируемость
5. Визуальные эффекты

---

# 7. Поведение при неясностях

Если задача:

- Не определена точно
- Может повлиять на архитектуру
- Требует изменения стека

ИИ‑Агент обязан:

- Уточнить требования
- Предложить 1–2 варианта
- Не внедрять изменения автоматически

---

# 8. Основной принцип проекта

Проект должен оставаться:

- Лёгким
- Управляемым
- Быстрым
- Понятным
- Без избыточной сложности

Любое решение должно соответствовать этим критериям.
